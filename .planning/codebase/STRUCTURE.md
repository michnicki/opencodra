## Directory Purposes

**`src/server/core/`:**
- Purpose: the review pipeline and other central business logic that doesn't belong to a single data table or external provider
- Contains: `review.ts` (phase orchestration), `diff.ts` (unified diff parsing), `github.ts` (low-level GitHub API client), `github-oauth.ts`/`oauth.ts`/`sessions.ts` (auth), `verify.ts` (webhook signature check), `model-output.ts` (LLM JSON validation/repair), `llm-crypto.ts` (encrypt/decrypt stored provider keys), `job-recovery.ts` (maintenance sweep), `telemetry.ts`, `token-tracker.ts`, `timeout.ts`, `config.ts` (repo config loading), `logger.ts`, `http.ts`, `updates-email.ts`
- Key files: `src/server/core/review.ts` (1538 lines — the largest and most central file in the repo)

**`src/server/db/`:**
- Purpose: all Postgres access; one module per domain/table, no ORM
- Contains: `jobs.ts`, `file-reviews.ts`, `repositories.ts`, `repo-configs.ts`, `model-configs.ts`, `app-settings.ts`, `stats.ts`, `webhook-deliveries.ts`, `client.ts` (connection context)
- Key files: `src/server/db/client.ts` (defines `runWithDb`/`getDb`/`queryRows`/`queryTransaction` used by every other db module)

**`src/server/models/`:**
- Purpose: provider-specific LLM adapters behind a shared interface
- Contains: `openai.ts`, `anthropic.ts`, `google.ts`, `cloudflare.ts`, `types.ts` (shared interface), `catalog.ts` (model metadata/availability), `limits.ts` (context/token limits per model)

**`src/server/routes/` and `routes/api/`:**
- Purpose: HTTP endpoint definitions grouped by concern
- Contains: `webhook.ts` (GitHub events), `auth.ts` (OAuth redirect/callback), `api/auth.ts`, `api/jobs.ts`, `api/repos.ts`, `api/stats.ts`, `api/models.ts`, `api/settings.ts`

**`src/server/workflows/`:**
- Purpose: Cloudflare Workflows entrypoint classes (durable execution)
- Contains: `review.ts` (`ReviewWorkflow`, the only workflow in the system)

**`src/client/components/features/`:**
- Purpose: components scoped to a specific dashboard feature/page rather than generic reusable UI
- Contains: subdirectories per feature — `dashboard/`, `job-detail/`, `models/`, `reviews/`, `stats/`

**`src/client/components/ui/`:**
- Purpose: generic Radix-based design-system primitives reused across all pages
- Contains: `button.tsx`, `card.tsx`, `dialog` (via `confirm-dialog.tsx`), `dropdown-menu.tsx`, `input.tsx`, `select.tsx`, `switch.tsx`, `alert.tsx`, `badge.tsx`

**`src/shared/`:**
- Purpose: single source of truth for data shapes crossing the worker/queue/DB-JSON/client boundary
- Contains: Zod schemas (`schema.ts`) plus plain types for API (`api.ts`), config defaults (`config.ts`), GitHub payloads (`github.ts`), and transient-error string matching (`transient-errors.ts`)

**`db/migrations/`:**
- Purpose: versioned, plain-SQL schema history, applied in numeric order under an advisory lock
- Contains: `001_initial.sql`, `002_jobs_async_review.sql` (naming: `NNN_description.sql`)

**`test/`:**
- Purpose: Vitest test suite against a real disposable Postgres database
- Contains: `*.spec.ts` files named after the feature under test (`diff.spec.ts`, `webhook-handling.spec.ts`, `chunk-concurrency.spec.ts`, `resumable-queue.spec.ts`, etc.), `setup.ts` (global test setup, Cloudflare binding mocks), `helpers.ts` (shared test utilities), `mocks/` and `browser/` subdirectories

## Key File Locations

**Entry Points:**
- `src/server/index.ts`: Worker `fetch`/`queue`/`scheduled` exports
- `src/client/main.tsx`: React SPA bootstrap
- `src/server/workflows/review.ts`: `ReviewWorkflow` durable-execution entrypoint

**Configuration:**
- `wrangler.jsonc`: Cloudflare bindings (Hyperdrive, KV, Queue, AI, Assets, Workflow)
- `src/server/env.ts`: TypeScript contract for all bindings/env vars (`AppBindings`)
- `vite.config.ts` / `tsconfig.json`: path aliases (`@server`, `@client`, `@shared`, `@`)
- `.dev.vars` (local, git-ignored, copy from `.dev.vars.example`): local secrets/bindings

**Core Logic:**
- `src/server/core/review.ts`: review job phase state machine — start here for any review-pipeline change
- `src/server/workflows/review.ts`: how phases map onto Cloudflare Workflow steps/sleeps
- `src/shared/schema.ts`: change data shapes here first, before touching implementation

**Testing:**
- `test/*.spec.ts`: one file per feature/scenario
- `test/setup.ts`: global fixtures and Cloudflare binding mocks (KV, Queue, ASSETS)
- `scripts/migrate.mjs`: applies `db/migrations/*.sql` to `TEST_DATABASE_URL` before `npm test` runs

## Naming Conventions

**Files:**
- kebab-case for all TypeScript/TSX files: `file-reviews.ts`, `job-detail.tsx`, `use-polling.ts`

**Directories:**
- lowercase, plural for collections of similar modules: `routes/`, `models/`, `hooks/`, `components/`
- singular/domain-named for cohesive subsystems: `core/`, `db/`

**Test files:**
- `<feature-name>.spec.ts`, colocated in the top-level `test/` directory (not co-located with source)

**React components:**
- One component per file, filename matches default export in kebab-case (e.g. `job-header.tsx` exports `JobHeader`)

**Hooks:**
- Prefixed `use-`: `use-polling.ts`, `use-job-detail.ts`, `use-is-dark-mode.ts`

## Where to Add New Code

**New review-pipeline behavior (e.g. a new phase or model routing rule):**
- Primary code: `src/server/core/review.ts` (phase logic) or `src/server/services/model.ts` (routing)
- Data shape changes: `src/shared/schema.ts` first
- Tests: new `test/<feature>.spec.ts`

**New LLM provider:**
- Implementation: new file in `src/server/models/` implementing the shared interface (`types.ts`), registered in `models/catalog.ts`
- Tests: `test/model-service.spec.ts` pattern, or a new `test/<provider>-model.spec.ts`

**New dashboard API endpoint:**
- Router: `src/server/routes/api/<domain>.ts`, registered in `src/server/app.ts`
- DB access: add/extend a module in `src/server/db/`
- Response shape: add to `src/shared/api.ts`
- Tests: `test/api.spec.ts` or a domain-specific spec

**New DB table/column:**
- Migration: new numbered file in `db/migrations/` (`00N_description.sql`)
- Access module: new or extended file in `src/server/db/`

**New dashboard UI feature:**
- Page: `src/client/pages/<name>.tsx`, registered with React Router
- Feature components: `src/client/components/features/<feature>/`
- Reusable primitives: `src/client/components/ui/` (only if truly generic/design-system level)
- Data fetching: extend `src/client/lib/api.ts`; use `src/client/hooks/use-polling.ts` for live-updating views

**Utilities:**
- Server-side shared helpers: `src/server/core/` (if business-logic-adjacent) — there is no generic `utils/` on the server
- Client-side shared helpers: `src/client/lib/utils.ts`

## Special Directories

**`db/migrations/`:**
- Purpose: schema version history
- Generated: No (hand-written SQL)
- Committed: Yes

**`src/server/worker-env.d.ts`:**
- Purpose: generated ambient types matching `wrangler.jsonc` bindings
- Generated: Yes, via `npm run cf-typegen` (also runs as part of `npm run build`)
- Committed: Yes (must be regenerated after changing bindings)

**`dist/`:**
- Purpose: Vite client build output served as Worker `ASSETS`
- Generated: Yes
- Committed: No

**`public/`:**
- Purpose: static files copied verbatim into the client build (icons, images)
- Generated: No
- Committed: Yes

**`.planning/`:**
- Purpose: GSD workflow planning artifacts (this document's own home)
- Generated: Partially (some files hand-written, some agent-generated)
- Committed: Project-dependent
