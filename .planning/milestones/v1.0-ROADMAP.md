# Roadmap: Codra — Bitbucket Cloud Support

## Overview

Codra reviews GitHub pull requests today; this milestone adds Bitbucket Cloud as a fully coexisting second provider. The build order is schema-first, then interface-first: the data model learns to represent either provider (Phase 1), the existing GitHub pipeline is wrapped behind a `VcsProvider` interface with zero behavior change and a regression-test safety net (Phase 2), and the webhook ingestion logic GitHub already uses is extracted into a shared, reusable module (Phase 3) — all three phases are exercised exclusively by the existing all-GitHub fleet, so GitHub is never at risk from new code paths. Only once that foundation is in place does Bitbucket-specific work begin: bot authentication and encrypted credential storage (Phase 4), then the Bitbucket adapter, webhook route, and review-posting logic (Phase 5), and finally the user-facing capstone — Bitbucket dashboard login and a provider-aware "add repo" flow (Phase 6). At every step, the full existing test suite must stay green and GitHub's behavior must be unchanged; this is a cross-cutting requirement (NREG-02) that applies to every phase, not a milestone owned by any one of them.

## Phase Details

### Phase 1: Schema & Contract Foundation

**Goal**: The data model can represent either VCS provider — additive only, zero risk to existing GitHub rows — before any provider-specific code is written.
**Depends on**: Nothing (first phase)
**Requirements**: FND-01, FND-02, FND-03, NREG-03
**Success Criteria** (what must be TRUE):

  1. Every existing repository row continues to behave identically after migration, now carrying `vcs_provider = 'github'` by default with no manual backfill step required.
  2. A repository can be inserted with `vcs_provider = 'bitbucket'` sharing the same `owner`/`repo` as an existing GitHub row without violating the uniqueness constraint.
  3. `jobs` rows can carry provider-agnostic `status_check_ref`/`review_ref` text values alongside the existing GitHub-shaped id columns, which remain populated and readable for existing GitHub jobs.
  4. A queue message enqueued by pre-migration code (no `provider` field present) still validates against the updated `reviewJobMessageSchema` and is treated as `'github'` — verified by an old-fixture-replay test.
  5. The full existing test suite passes unmodified after the migration and schema change.

### Phase 2: VCS Provider Abstraction

**Goal**: The existing GitHub review pipeline runs through a provider-agnostic `VcsProvider` interface with zero behavior change, guarded by regression tests written before the refactor lands.
**Depends on**: Phase 1
**Requirements**: FND-04, FND-05, NREG-01

  4. Every existing GitHub review scenario — prepare/review/finalize, retries, double-post guard, supersede-on-new-push — passes unchanged through the new branch point, with the same regression tests from criterion 1 still green.

### Phase 4: Bitbucket Bot Auth & Credential Storage

**Goal**: Codra can securely hold, and the dashboard can surface, a valid Bitbucket bot credential per repo — with expiry visible from day one — before any Bitbucket API call is made.
**Depends on**: Phase 1 (schema), Phase 3 (not a hard code dependency, but sequenced after the GitHub-safe foundation per the research build order)
**Requirements**: AUTH-01, AUTH-02, AUTH-03
**Success Criteria** (what must be TRUE):

  1. An operator can store a Bitbucket Repository/Workspace Access Token (Bearer) and a per-repo webhook secret for a repo; both are persisted encrypted in Postgres (AES-GCM, mirroring `core/llm-crypto.ts`) and never appear in plaintext in logs or API responses.
  2. Bot authentication uses only the stored access token — no app-password flow and no Atlassian Connect app/JWT lifecycle exist anywhere in the code path.
  3. The dashboard displays each Bitbucket credential's status, including its expiry date when the token has one, without requiring a live Bitbucket API call on every page load.
  4. Replacing/rotating a stored token or webhook secret updates the encrypted value in place, with no schema change or manual DB edit required.

### Phase 5: Bitbucket Adapter & Webhook Route

**Goal**: A Bitbucket Cloud pull request receives the same automated review — webhook receipt, diff fetch, inline findings, summary, and merge-gating status — that a GitHub PR already gets, without touching GitHub's live path.
**Depends on**: Phase 2 (interface), Phase 3 (shared ingest), Phase 4 (credentials)
**Requirements**: BB-01, BB-02, BB-03, BB-04, BB-05, REV-01, REV-02, REV-03
**Success Criteria** (what must be TRUE):

  1. A `pullrequest:created` webhook for a Bitbucket repo with a valid `X-Hub-Signature` is accepted at `POST /webhook/bitbucket` and enqueues a review job; a request with a missing or invalid signature is rejected before any job is created (fail closed).
  2. A `pullrequest:updated` event whose `source.commit.hash` is unchanged from the last-processed value does not trigger a re-review; a event where the commit hash has changed does.
  3. PR diff and diffstat are fetched into the existing `core/diff.ts` pipeline for representative Bitbucket fixtures (plain edits, renamed files, binary files), respecting Bitbucket's 200-file/8,000-line diff caps and staying within the Workers per-invocation subrequest budget on a large synthetic PR.
  4. Inline findings are posted on the Bitbucket PR anchored by file `path` and `to`/`from` line (no `commit_id` dependency); a Code Insights report reflects PASSED/FAILED with a summary; a commit build status reflects SUCCESSFUL/FAILED — all three keyed so a retried job updates them in place rather than duplicating.
  5. None of the above changes GitHub's existing webhook route, adapter, or posting behavior — the full test suite, including the GitHub end-to-end pipeline test, remains green throughout.

### Phase 6: Bitbucket Dashboard Login & Repo Onboarding UX

**Goal**: A user can log into the Codra dashboard using Bitbucket and add a Bitbucket repo end-to-end through the UI, at parity with the existing GitHub flow.
**Depends on**: Phase 4 (credential model), Phase 5 (a repo must be onboardable to be reviewable)
**Requirements**: DASH-01, DASH-02
**Success Criteria** (what must be TRUE):

  1. A user can log into the dashboard via Bitbucket OAuth (authorization_code grant); the resulting session/allow-list check is keyed on `account_id`/`uuid`, never on `username`.
  2. The "add repo" flow presents a provider picker (github | bitbucket); choosing Bitbucket surfaces a form to enter an access token and webhook secret.
  3. After submitting the form, the new Bitbucket repo appears in the dashboard's repo list and is immediately eligible to receive webhook-triggered reviews.
  4. The existing GitHub dashboard login and "add repo" flow are visually and behaviorally unaffected by the new provider picker.
