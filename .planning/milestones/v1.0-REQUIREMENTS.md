# Requirements: Codra — Bitbucket Cloud Support

**Defined:** 2026-07-12
**Core Value:** A Bitbucket Cloud PR receives the same automated AI review (inline findings posted back) that a GitHub PR already gets, from one Codra instance, without breaking existing GitHub support.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases (see Traceability). Scope = full GitHub parity for Bitbucket Cloud, table stakes only (differentiators deferred to v2).

### Foundation

- [x] **FND-01**: `repositories` table gains `vcs_provider` (default `'github'`, existing rows backfilled); `UNIQUE(vcs_provider, owner, repo)` replaces `UNIQUE(owner, repo)`
- [x] **FND-02**: `jobs` gains provider-agnostic `TEXT` reference columns (`status_check_ref`, `review_ref`) instead of GitHub-shaped `BIGINT` ids
- [x] **FND-03**: An optional, defaulted `provider` discriminator is added to `reviewJobMessageSchema` / repo config in `src/shared/schema.ts` (contract-first)
- [x] **FND-04**: A `VcsProvider` interface is defined and the existing GitHub client/service is wrapped behind a GitHub adapter with zero behavior change
- [x] **FND-05**: `core/review.ts` dispatches VCS operations through a single `VcsService` branch point (GitHub-only at this stage)
- [x] **FND-06**: Webhook dedup/insert/supersede/enqueue logic is extracted into a shared `core/webhook-ingest.ts`, consumed unchanged by the existing GitHub `/webhook` route

### Authentication & Credentials

- [x] **AUTH-01**: Codra authenticates as the Bitbucket review bot via a Repository/Workspace Access Token (Bearer) — not app passwords, not Atlassian Connect
- [x] **AUTH-02**: Bitbucket access tokens and per-repo webhook secrets are stored encrypted in Postgres (AES-GCM, mirroring `core/llm-crypto.ts`)
- [x] **AUTH-03**: The dashboard surfaces Bitbucket credential status, including expiry where applicable (credentials are not "set once and forget")

### Bitbucket Integration

- [x] **BB-01**: A hand-rolled `BitbucketClient` (`core/bitbucket.ts`) calls `api.bitbucket.org/2.0` via `fetch`, mirroring `GitHubClient` (no SDK)
- [x] **BB-02**: `POST /webhook/bitbucket` ingests `pullrequest:created` and `pullrequest:updated` events
- [x] **BB-03**: Bitbucket webhooks are verified via `X-Hub-Signature` HMAC using the per-repo secret (repo identified from payload before verification); requests with missing/invalid signatures fail closed
- [x] **BB-04**: Re-review is triggered by a change in `source.commit.hash`, so metadata-only `pullrequest:updated` events do not re-run a review
- [x] **BB-05**: PR diff + diffstat are fetched into the existing `core/diff.ts` pipeline, with chunking/budget logic generalized for Bitbucket's 200-file / 8,000-line diff caps and ~1,000 req/hr rate limit

### Review Output

- [x] **REV-01**: Inline findings are posted onto the Bitbucket PR anchored by file `path` + `to`/`from` line (no `commit_id` dependency)
- [x] **REV-02**: A Code Insights report (PASSED/FAILED + summary) is posted, keyed so retries update it in place
- [x] **REV-03**: A commit build status (SUCCESSFUL/FAILED) is posted for merge-gating, keyed by a stable `key`

### Dashboard

- [x] **DASH-01**: Dashboard login supports Bitbucket via a separate OAuth 2.0 consumer (authorization_code); the allow-list keys on `account_id`/`uuid` (never `username`)
- [x] **DASH-02**: The "add repo" flow has a provider picker (github | bitbucket) and a Bitbucket access-token + webhook-secret entry form

### Non-Regression

- [x] **NREG-01**: Regression tests for the GitHub state-machine invariants (lease/heartbeat, `forceFreshInstance` threading, supersede-on-new-push) are added *before* the abstraction refactor
- [x] **NREG-02**: The full existing test suite stays green after every phase; existing GitHub review behavior is unchanged
- [x] **NREG-03**: The schema migration is additive (defaults/backfill) and deployed so in-flight GitHub jobs are never dropped as malformed (verified by an old-fixture replay test)

## v2 Requirements

Deferred to a future release. Tracked but not in the current roadmap.

### Differentiators

- **ANNO-01**: Per-line Code Insights annotations (diff-gutter markers) mirroring inline comments
- **WS-01**: Workspace-level token/webhook covering many repos to reduce per-repo onboarding friction

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Bitbucket Data Center / Server | Different REST API dialect and auth model; Cloud only for this milestone |
| Atlassian Connect app model | End-of-support 2026-12, no new registrations since Feb 2026; overweight for single-tenant self-hosted |
| App passwords for bot auth | Fully removed by Atlassian on 2026-07-28 |
| Replacing / deprecating GitHub | GitHub and Bitbucket must coexist in one instance |
| GitLab, Azure DevOps, other VCS providers | Not in scope for this milestone |
| Bitbucket Pipelines / CI integration | Not part of the PR-review use case |
